# Goal 520: Claude External Review
Date: 2026-04-17
Reviewer: Claude Sonnet 4.6

## Verdict: APPROVE

The three Stage-1 proximity apps honestly implement currently supportable workloads
using existing RTDL kernels and Python orchestration without overstating language or
performance claims.

---

## Per-App Assessment

### ANN Candidate Search (`rtdl_ann_candidate_app.py`)

**Kernel used:** `knn_rows(k=1)` over a Python-supplied candidate subset.

**RTDL/Python boundary:** Correctly drawn. RTDL performs kNN reranking over the
candidate set; Python owns candidate selection, exact brute-force comparison,
and recall/distance-ratio evaluation.

**Honesty check:**
- The app is deliberately designed so recall_at_1 = 2/3, not 1.0. This is an honest
  portrayal of approximation degradation when the candidate set excludes the true
  nearest neighbor.
- The embedded `boundary` field reads: "RTDL does not yet provide an ANN index,
  training phase, or recall/latency optimizer." This is accurate and explicit.
- The candidate subset in the fixture is statically hand-selected rather than
  generated by a real approximate-search algorithm (LSH, HNSW, etc.). This is a
  known limitation and is consistent with the stated scope: the app demonstrates the
  candidate-reranking step of ANN, not a full ANN pipeline. The label "ANN candidate
  search" is accurate under this reading.

**No overstatement found.**

---

### Outlier Detection (`rtdl_outlier_detection_app.py`)

**Kernel used:** `fixed_radius_neighbors(radius=0.35, k_max=16)`.

**RTDL/Python boundary:** RTDL emits neighbor rows; Python counts density and applies
the outlier threshold. The boundary is implemented correctly.

**Honesty check:**
- `matches_oracle: true` is verified by a brute-force Python reference on every run.
- The `boundary` field reads: "Bounded density-threshold outlier demo only; RTDL does
  not yet expose density scoring as a language primitive." This is accurate.
- `k_max=16` is a neighbor-count cap that holds for the small portable fixture. The
  cap is not hidden; it appears as a named constant and a top-level output field.
- No performance claims are made. Default backend is `cpu_python_reference`.

**No overstatement found.**

---

### DBSCAN Clustering (`rtdl_dbscan_clustering_app.py`)

**Kernel used:** `fixed_radius_neighbors(radius=0.35, k_max=16)` (same kernel as
outlier detection).

**RTDL/Python boundary:** RTDL emits neighbor rows; Python implements core/border/noise
classification and BFS-based density-connected cluster expansion. This is the standard
DBSCAN algorithm expressed correctly at the Python layer.

**Honesty check:**
- `matches_oracle: true` is verified by a brute-force Python reference on every run.
- `copies=2` test verifies cluster isolation across copies (x_offset=6.0 >> EPSILON=0.35),
  confirming no cross-copy boundary leakage.
- The `boundary` field reads: "Bounded app-level DBSCAN demo only; RTDL does not yet
  expose clustering expansion or connected-component reduction as language primitives."
  This is accurate.
- DBSCAN expansion, label propagation, and connected-component logic are entirely in
  Python; RTDL is not credited for them.

**No overstatement found.**

## Cross-Cutting Observations

1. **No new RTDL language internals were introduced.** Both `knn_rows` and
   `fixed_radius_neighbors` are existing primitives. This is consistent with the
   Goal520 report.

2. **No performance claims.** All three apps default to `cpu_python_reference` and
   make no statements about latency, throughput, or acceleration advantage.

3. **Boundary language is embedded in the artifact.** Each `run_app()` return value
   includes both an `rtdl_role` and a `boundary` field. Tests assert these fields are
   present and contain the expected disclaim text.

4. **Oracle parity is tested, not just asserted.** Both the outlier and DBSCAN apps
   compare against a brute-force Python reference at runtime. The ANN app reports
   recall and distance ratio, which are honest approximation metrics.

5. **Test coverage is adequate for the stated scope.** `tests/goal520_dbscan_clustering_app_test.py`
   covers all three apps: CLI JSON emission, oracle parity, scaling, boundary message
   preservation, and public-doc presence. The 22-test suite passes.

---

## Minor Observation (non-blocking)

The "ANN candidate search" label is the most aspirational name of the three. The app
demonstrates the reranking half of an ANN pipeline (coarse candidate selection →
exact-distance reranking within the candidate set) but the coarse selection step is a
static hand-authored fixture rather than a live approximate index. This is accurately
disclaimed in the `boundary` field. No change required, but future Stage-2 work that
adds dynamic candidate generation (e.g., grid-bucketing, random-projection hashing)
would materially strengthen the "ANN" framing.

---

## Summary

All three apps implement what they claim using currently available RTDL + Python
surfaces. Language and performance claims are within bounds. Oracle parity is tested.
Boundaries are stated in code, tests, and public docs. Goal 520 is approved for merge.
