### F1 — CRITICAL: Section 5.6 published report is not reproducible from the documented command, and contains misleading pip claim

**Evidence:**

- `docs/reports/section_5_6_scalability_report_2026-03-31.md` states `R=100,000`, probe series `100K–500K`, 5 iterations.
- `src/rtdsl/section_5_6_scalability.py` code defaults: `FIXED_BUILD_POLYGONS = 800`, argparse defaults `--build-polygons 800 --probe-series 500,1000,1500,2000,2500`.
- `build/section_5_6_scalability/section_5_6_scalability.json` — the current build artifact — has `R=800`, probe series `160–800`, 2 iterations.
- `docs/reports/rtdl_embree_paper_report_2026-03-31.md` (a related report) says explicitly `R = 800`, matching the build artifact, directly contradicting the docs section_5_6 report.
- `make eval-section-5-6` runs `python3 -m rtdsl.section_5_6_scalability` with no arguments → produces the 800-polygon version → with `publish_docs=True` default, **would overwrite the published 100K report**, destroying the claimed result.
- The 100K report was generated by running with undocumented CLI flags (`--build-polygons 100000 --probe-series 100000,200000,300000,400000,500000 --workloads lsi`), nowhere recorded in README or Makefile.
- That run used `--workloads lsi` only: `pip` parity is `None` in the published report, meaning no pip correctness check was done. Yet the report's Interpretation section (boilerplate) says *"Figure 14 analogue corresponds to pip only"* and the Results table has no pip rows. Figure 14 SVG is an empty placeholder.

**Impact:** The Section 5.6 report cannot be reproduced by any documented command; its pip claim is a false positive (Figure 14 is empty); and running the README command would silently destroy the published 100K numbers.

### F2 — HIGH: "Close to native" performance claim omits that "native" shares the same underlying library

**Evidence:**

- `apps/goal15_lsi_native.cpp` and `apps/goal15_pip_native.cpp` call `rtdl_embree_run_lsi()` and `rtdl_embree_run_pip()` directly from `librtdl_embree.dylib` — the exact same C entry points that RTDL's Python path calls via ctypes.
- Goal 15 report explicitly states: *"the native executables currently call the same Embree C API implementation already exposed by RTDL."*
- README performance summary: *"the raw and prepared-raw RTDL Embree paths are close to native"* and *"lsi: raw 0.98x ... pip: raw 0.87x gap vs native"* — RTDL is actually **faster** than "native" at larger profiles in absolute terms.
- The word "native" to an external reader means an independently written C++ implementation. Here it means a thin C++ wrapper around the identical shared library function, measuring only Python/ctypes packing overhead.

**Impact:** The performance claim as written implies competitive parity against a standalone C++ algorithm. The actual comparison is narrower: it measures Python binding and data-prep overhead against a direct C API call into the same compiled code. The "close to native" framing is technically defensible only if the comparison boundary is clearly stated; the README omits it.

### F3 — MEDIUM: Goal 18 "raw gap vs native lower-bound" comparisons are methodologically invalid

**Evidence:**

- Goal 18 report compares `raw total: 0.000093375 s` (RTDL, 2-row dataset, hot path) against `Goal 15 native lower-bound: 0.000965458 s` (a full native process run: startup + CSV I/O + Embree call + output for a larger fixture).
- Results: `raw gap vs native lower-bound: 0.10x` and `prepared raw gap vs native lower-bound: 0.01x`.
- These are not comparing the same thing — RTDL is timed in-process on a trivial dataset, native is an end-to-end process execution on a different fixture.

**Impact:** The 0.10x and 0.01x figures are meaningless as performance comparisons and could mislead a reader into thinking RTDL is 10–100x faster than native, which is the opposite of the actual measured relationship on real workloads.

### F4 — MEDIUM: `segment_polygon_hitcount` and `point_nearest_segment` listed under "packed/prepared Embree coverage" but are nested loops

**Evidence:**

- `src/native/rtdl_embree.cpp:936–976` and `978+`: both `rtdl_embree_run_segment_polygon_hitcount` and `rtdl_embree_run_point_nearest_segment` are pure nested C++ for-loops with zero Embree API calls.

**Impact:** Calling these workloads "Embree coverage" directly contradicts the semantic boundaries section in the same document. A reader building on the runtime slice section alone would believe these workloads use Embree BVH acceleration.

## Suggested Revisions

**R1 (for F1 — Critical):** Add the exact command used to generate the published 100K section_5_6 report to README and Makefile as a named target (e.g., `eval-section-5-6-large`). Separate it from the default `eval-section-5-6`. Fix the `publish_docs=True` mechanism so the default command does not silently overwrite the larger-scale published report. Add a conditional to the Interpretation template so it doesn't emit "Figure 14 analogue corresponds to pip only" when pip is not in the workloads. **R3 (for F3 — Medium):** Remove or flag the `raw gap vs native lower-bound: 0.10x` / `0.01x` values in the Goal 18 report as methodologically incomparable. If a valid micro-benchmark comparison is desired, run both paths on identical in-process workloads.

**R4 (for F4 — Medium):** In the "Current low-overhead runtime slice" section of README, add a parenthetical after `segment_polygon_hitcount` and `point_nearest_segment` noting that these use `native_loop` not BVH — consistent with the semantic boundaries section below.

**R5 (for F5 — Low):** Delete the duplicate `docs/vision.md` line in the Repository Layout section.

**R7 (for F7 — Low):** Replace the absolute artifact paths in `goal15_cpp_embree_comparison_2026-03-31.md` with repo-relative paths.

**R8 (for F8 — Low):** Set a minimum meaningful scale for the default `eval-section-5-6` run (e.g., at least a few hundred polygons where intersections occur) or document clearly that the 800-polygon default is a smoke-test configuration, not a performance demonstration.
