## Goal 15 Pre-Action Plan

### 1. Exact Minimal Native Programs

Two standalone C++ executables, no RTDL involvement:

**`bench/lsi_native.cpp`**
- Reads N left segments and M right segments from a binary flat file (`float32 x0 y0 x1 y1` per segment, little-endian).
- Builds an Embree BVH over the right segments using fat rays (zero-thickness).
- Queries each left segment as a ray to collect candidate pairs.
- Refines each candidate with an exact segment-segment intersection predicate (Cohen-Sutherland or Shamos-Hoey; no Embree involvement in the refinement step).
- Writes the accepted pairs to stdout as sorted ASCII CSV (see §2).

**`bench/pip_native.cpp`**
- Reads N query points and M polygons. Polygons are stored as triangle fans (each polygon = one or more triangles covering its interior).
- Builds an Embree BVH over the polygon triangles.
- For each query point, shoots a +X axis ray and counts hits using Embree's `intersect` (not `occluded`), accumulating a crossing count; point-in-polygon result = `crossing_count % 2`.
- Writes the accepted (point_id, polygon_id) pairs as sorted ASCII CSV.

No other executables in the first scope. The RTDL+Embree path already exists or will be generated by the compiler; its output uses the same file format.

---

### 2. Exact Output Artifact Format

Both programs write to stdout in the same format so diffing is trivial:

```text
left_id,right_id
0,17
0,42
1,3
...
```

Rules:
- Integer IDs only (no floats) in the result file. Intersection coordinates are **not** in the comparison file.
- Sorted ascending by `left_id`, then by `right_id` (or `point_id` then `polygon_id` for pip).
- No header line.
- One pair per line, Unix line endings.
- A separate optional `_coords` file can hold float coordinates for manual inspection but is **not** used in the automated correctness check.

---

### 3. Exact Correctness Comparison Method

**Reference oracle**: a Python script `bench/brute_force_lsi.py` (and `_pip.py`) that checks every pair in O(N×M) using exact rational arithmetic via Python's `fractions.Fraction`. This is slow but unambiguously correct.

**Comparison steps**:
1. Run oracle → `ref.csv`.
2. Run `lsi_native` → `native.csv`.
3. Run RTDL+Embree path → `rtdl.csv`.
4. `diff ref.csv native.csv` — must be **exact byte-for-byte match** (pair IDs are integers; no float comparison here).
5. `diff ref.csv rtdl.csv` — same requirement.
6. `diff native.csv rtdl.csv` — redundant sanity check.

**What uses float comparison and what does not**:
- Pair membership (which IDs intersect) — **exact set equality**, no tolerance.
- Intersection point coordinates (if ever logged) — **not checked** in correctness gate; they go in the `_coords` file for human review only. If they must be checked later, tolerance = `1e-9` absolute (double precision).

There is no "close enough" on pair membership. A missed or spurious pair is a correctness failure regardless of magnitude.

---

### 4. Exact Timing Methodology

**What is timed** (inside `TIMER_START` / `TIMER_STOP` brackets):
- BVH build over the static side.
- Query phase: ray casting + Embree traversal + refinement predicate evaluation.

**What is NOT timed**:
- Reading input from disk.
- Allocating input buffers and filling them.
- Writing output to disk.
- Embree device/scene creation and destruction overhead outside the build call.
- The sort step for output (correctness artifact only, not in hot path).

**Timer**:
- `clock_gettime(CLOCK_MONOTONIC_RAW)` on Linux / `mach_absolute_time()` on macOS, reported in microseconds.
- Report BVH build time and query time as **separate** numbers.

**Protocol**:
1. One untimed warm-up run (to prime instruction cache and OS page tables).
2. Five timed runs in the same process invocation using the same pre-loaded input buffers.
3. Report the **median** of the five runs. Do not report mean or min.
4. Input sizes tested: 10 K × 10 K pairs, 100 K × 100 K pairs, 1 M × 1 M pairs (or the largest that fits in available RAM).

Both `lsi_native` and the RTDL+Embree path use the identical timing wrapper so the boundary of what is timed is the same by construction, not by convention.

### 5. Exact First Implementation Slice

**Build `bench/lsi_native.cpp` for a synthetic 1 K × 1 K dataset only.**

Concretely:
- Input: a hardcoded generator inside `main()` that fills two arrays of 1000 segments each on a unit square grid using a fixed seed (no file I/O in this slice).
- No timing code in this slice.
- Output: sorted `left_id,right_id` CSV to stdout.
- Verify with `bench/brute_force_lsi.py` on the same 1 K × 1 K synthetic data.
- Gate: `diff <(./lsi_native) <(python3 brute_force_lsi.py)` must produce no output.

Nothing else is built until this gate passes. File I/O, timing, `pip_native`, and RTDL+Embree integration come in subsequent slices.
