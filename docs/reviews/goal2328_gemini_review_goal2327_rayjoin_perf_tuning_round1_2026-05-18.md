# Independent Gemini Review: Goal2327 RayJoin Perf Tuning Round 1

**Date:** 2026-05-18

This is an independent Gemini review, distinct from Codex, for Goal2327.

## Review Questions and Answers:

### 1. Does the new prepared OptiX route advance the six requested RayJoin work tracks: generic device-resident row stream / continuation, generic grouped count/parity reduction, stronger prepared closed-shape membership, many-query batching, phase-separated timing, and paper-protocol discipline?
Yes, the new prepared OptiX route advances several of the requested work tracks:
*   **Generic device-resident row stream / continuation:** Partially advanced. The design boundary is recorded and the RayJoin app explicitly exposes the missing status, indicating a clear understanding of this gap and future work. However, full implementation of a generic device-resident row stream continuation is still pending native work.
*   **Generic grouped count/parity reduction:** Partially advanced. Existing partner reductions are identified as the CPU/GPU partner layer, but native continuation for this is still pending.
*   **Stronger prepared closed-shape membership:** Advanced. This is explicitly exposed through the benchmark app's prepared OptiX route, using `prepare_point_closed_shape_membership_2d_optix`.
*   **Many-query batching and launch grouping:** Advanced. The prepared route and pod runner incorporate warmups and repeats, and same-query stream replay functionality is available.
*   **Phase-separated timing:** Advanced. This is implemented and reported within `run_rayjoin_prepared_optix_workload(...)` via `phases_sec` and `native_phase_timings`.
*   **Paper-protocol reproduction discipline:** Advanced. The pod runner reuses the existing same-query stream harness and maintains the necessary locked claim boundaries.

### 2. Does the change preserve the RTDL rule that native engines remain app-agnostic, with RayJoin interpretation staying in Python/app code?
Yes, the change preserves the RTDL rule that native engines remain app-agnostic. The `rtdl_rayjoin_v2_spatial_join_app.py` clearly states its `native_engine_boundary` as seeing "generic prepared point/closed-shape or segment-pair contracts," with "RayJoin application policy and paper-specific interpretation stay[ing] in Python." The `README.md` and `docs/research/future_version_to_do_list.md` also consistently reinforce that native acceleration should use generic traversal, not RayJoin-specific native kernels, and that application-specific logic remains in Python.

### 3. Are the claim boundaries still appropriately locked against public RayJoin-vs-RTDL speedup claims, broad RT-core claims, true-zero-copy claims, and v2.0 release authorization?
Yes, the claim boundaries are explicitly and appropriately locked. The `docs/reports/goal2327_rayjoin_perf_tuning_round1_2026-05-18.md` clearly lists the claims that are *not* authorized by this round, including RTDL beating RayJoin, full RayJoin paper reproduction, broad RT-core speedup, whole-application speedup, true zero-copy, and v2.0 release authorization. These boundaries are consistently reflected in the `claim_boundary` payload within `rtdl_rayjoin_v2_spatial_join_app.py` and the `claim_boundary.json` output generated by `scripts/goal2327_rayjoin_pod_perf_runner.sh`, as well as prominently documented in the `README.md`.

### 4. Is the pod runner safe and useful for a long RTX timing round, including visible progress, timeout control, OptiX library precondition, optional same-query stream replay, and clear artifact output paths?
Yes, the pod runner (`scripts/goal2327_rayjoin_pod_perf_runner.sh`) appears safe and useful for its intended purpose:
*   **Visible progress:** The script uses a `log` function to output "START" and "DONE" messages for each step, indicating clear progress during execution.
*   **Timeout control:** Configurable `STEP_TIMEOUT_SECONDS` is enforced using the `timeout` command for each major step, preventing indefinite runs.
*   **OptiX library precondition:** The script includes an explicit check for the `RTDL_OPTIX_LIBRARY` environment variable at startup and exits if it's not set, ensuring the necessary OptiX environment.
*   **Optional same-query stream replay:** The runner supports optional stream replay via `LSI_STREAM` and `PIP_STREAM` environment variables, utilizing `scripts/goal2292_rayjoin_current_prepared_comparison.py` for this purpose. It correctly logs when this step is skipped if streams are not provided.
*   **Clear artifact output paths:** The `OUTPUT_DIR` variable is well-defined and used to store organized JSON reports, and the script logs the final output location.

### 5. Are there risks in the route split, PIP/LSI-only support, or overlay exclusion that should be documented before the pod run?
The primary risk identified and documented is the deliberate exclusion of overlay support from the `prepared_optix` route. This is explicitly stated in `docs/reports/goal2327_rayjoin_perf_tuning_round1_2026-05-18.md`, `rtdl_rayjoin_v2_spatial_join_app.py`, and `README.md`. The reason for this exclusion is the current lack of a generic device-resident continuation or dependency-row stream, which overlay functionality requires. The documentation also explicitly warns against reintroducing RayJoin-specific native kernels to address such gaps (`docs/research/future_version_to_do_list.md`). The PIP/LSI-only support for the prepared OptiX route is a known and documented scope limitation, not a hidden risk. The clear documentation of this limitation and its rationale mitigates potential issues.

## Verdict:
`accept-with-boundary`
