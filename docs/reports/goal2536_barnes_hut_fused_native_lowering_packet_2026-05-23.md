# Goal2536 Barnes-Hut Fused Native Lowering Packet

Date: 2026-05-23

## Decision

The next Barnes-Hut engine target should be a fused generic operation, not a
Barnes-Hut-specific native app:

`aggregate-tree opening frontier -> weighted vector contribution -> grouped vector sum`

The reference contract already exists locally as:

- tree: `generic_bucketized_aggregate_tree_2d_v1`
- frontier: `generic_aggregate_tree_opening_frontier_2d_v1`
- materialized contributions:
  `generic_weighted_inverse_square_contribution_rows_2d_v1`
- materialized vector sum: `generic_grouped_vector_sum_rows_2d_v1`
- streamed vector sum:
  `generic_weighted_inverse_square_vector_sum_2d_v1`
- pressure guard:
  `generic_vector_sum_materialization_pressure_2d_v1`

The native/partner target is to implement the streamed semantics without
materializing frontier or contribution rows in Python.

## Proposed Fused Contract

Name:

- `generic_aggregate_frontier_weighted_vector_sum_2d_v1`

Inputs:

- weighted source points: `id`, `x`, `y`, `mass`
- aggregate tree nodes in prepared arrays:
  `id`, `cx`, `cy`, `half_size`, `mass`, `member_begin`, `member_count`,
  `child_begin`, `child_count`, `resume_index`, `is_leaf`
- packed leaf member IDs
- packed child IDs or first-child/count layout
- scalar `theta`
- scalar `softening`
- optional source range or batch descriptor

Outputs:

- per-source `vector_x`
- per-source `vector_y`
- optional per-source counters:
  `visited_node_count`, `accepted_aggregate_count`,
  `exact_contribution_count`
- metadata:
  `materialized_frontier_rows=false`,
  `materialized_contribution_rows=false`,
  `native_engine_app_specific=false`

Required semantics:

- Equivalent to `sum_weighted_inverse_square_contributions_2d(...)` when the
  same `opening_frontier` is generated by
  `evaluate_aggregate_tree_opening_frontier_2d(...)`.
- A node is accepted only when it does not contain the source and
  `(2 * half_size / distance) < theta`.
- If a node is not accepted and has children, traversal continues to children.
- If a non-accepted leaf is reached, exact member contributions are summed,
  excluding the source ID.
- The operation is vector-valued and grouped by source.

## CPU/Embree Lowering

Embree is not automatically a win for the full fused operation because the
opening rule is a tree-continuation algorithm, not a simple point-to-primitive
query. The first CPU lowering should therefore be:

- a prepared app-agnostic aggregate-tree descriptor;
- task-parallel per-source traversal;
- direct vector accumulation into output arrays;
- no Python frontier or contribution rows;
- optional Embree use only for specific spatial discovery substeps if it
  reduces traversal work without changing semantics.

This should be treated as a CPU-native fallback, not as an Embree speedup claim
until measured.

## OptiX Lowering

The authors' artifact suggests the OptiX direction:

- encode aggregate nodes as RT primitives;
- traverse in DFS/resume order;
- use first-hit traversal with disabled AnyHit;
- use hit/miss continuation metadata equivalent to `resume_index`;
- accumulate vector contributions on device;
- keep body/tree arrays device-resident across launches.

RTDL must keep this generic:

- no `barnes_hut` symbols in native ABI;
- no paper-specific dataset assumptions;
- no app-specific force law beyond the named generic weighted inverse-square
  vector-sum contract.

## Partner Lowering

The first partner path can be useful even before OptiX:

- CuPy/Torch custom kernel over prepared tree arrays;
- one source per thread or warp/block depending on tree size;
- explicit stack or rope/resume loop;
- direct writes to `vector_x/vector_y`;
- no host-side contribution-row materialization.

This is likely easier to validate than OptiX and can become the conformance
oracle for native OptiX.

## Validation Gates

Correctness gates:

- compare vector sums against `streamed_force_sum_bucketized_cpu`;
- compare small cases against exact all-pairs force;
- verify zero checksum deltas versus materialized contribution rows;
- verify source IDs with no contributions still produce zero vectors;
- verify source exclusion inside leaf exact fallback.

Performance gates:

- record Python reference timings separately from native/partner timings;
- record whether frontier/contribution rows were materialized;
- record prepared tree build time separately from per-step force time;
- no public speedup wording without same-contract input, commit, hardware, and
  external review.

## Stop Condition

At this point, the remaining high-value implementation work needs either:

- a CUDA/OptiX pod for native RT traversal, or
- a deliberate local decision to add 3-D reference semantics before native work.

Without that decision, the benchmark app is locally promoted and the next performance target is no longer blocked by missing Python reference contracts.
