# Detailed Code and Doc Audit: v0.5+v0.6 Sequences

## Audit Status: REMEDIATED

This audit report documents the technical flaws identified during the formal review of the `v0.6` Graph Workload line and the stabilization of the `v0.5` release branch.

## High-Priority Findings & Remediation

### 1. Python 3.9 Compatibility Regression
- **Issue**: `from __future__ import annotations` was placed after other code in `src/rtdsl/rtnn_manifests.py`, causing a `SyntaxError` on Python 3.9.
- **Remediation**: Moved the future import to the first line of the module.
- **Implication**: Previous `v0.5.0` claims of Python 3.9 support were technically broken on this branch.

### 2. Graph Boundary Failures (Large Vertex IDs)
- **Issue**: The `v0.6` truth-path logic failed on graphs with large vertex IDs (approaching $2^{32}$).
  - Python: `bfs_levels_cpu` attempted to allocate gigabytes of memory for a fixed-size `visited` list, causing OOM.
  - Native: The C++ oracle truncated 32-bit IDs or failed strictly-ascending checks due to signed/unsigned mismatches or 22-bit assumptions in the test suite.
- **Remediation**:
  - Migrated Python reference to use `set` for visited nodes.
  - Migrated Native Oracle to use `std::unordered_set`.
  - Updated tests to explicitly support 32-bit boundary vertex counts.

### 3. Partial CSR Memory Safety (REMEDIATED)
- **Issue**: The implementation assumed "Strict CSR" ($len = V+1$), which is memory-prohibitive for large sparse graphs. Attempts to use Partial CSR resulted in `Segmentation Fault` in the C++ oracle.
- **Remediation**:
  - Modified `csr_graph` to allow shorter offset arrays (Partial CSR).
  - Added explicit range checks in the C++ oracle loops to safely handle implicit sink vertices.

### 4. Performance Measurement Methodology
- **Issue**: The evaluation harness was suspected of capturing DDL/DML setup overhead (PostgreSQL index creation) within execution timing loops.
- **Remediation**: Verified `src/rtdsl/graph_eval.py`. The current implementation in `_timed_postgresql_call` successfully separates `prepare_fn` from `query_fn`. Timing results are confirmed to be valid (pure query execution).

### 5. Build System Parity (macOS Apple Silicon)
- **Issue**: Native oracle build failed on macOS due to missing search paths for Homebrew-installed GEOS libraries.
- **Remediation**: Added `/opt/homebrew/include` and `/opt/homebrew/lib` to the `oracle_runtime.py` build flags.

### 5. DuckDB Fallback Baseline (MISSING)
- **Issue**: The `v0.5` goals specified adding DuckDB as a fallback baseline for graph workloads. Our audit confirms that `duckdb` is completely missing from `external_baselines.py` and the evaluation harness.
- **Recommendation**: DuckDB support should be prioritized in the next `v0.6` increment to fulfill the cross-platform benchmarking requirements defined in the `v0.5` roadmap.

## Process Integrity Verdict

The `v0.6` sequence suffered from a "rush to implementation" that bypassed the data contract phase. By remediating the technical flaws in the truth paths and formally documenting the **Partial CSR Contract**, we have restored the integrity of the research line.
