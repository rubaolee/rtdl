# Goal 1565: OptiX collect-k fused materialize+mark diagnostic

## Verdict

The diagnostic fused materialize+mark kernel is correct on the controlled pod
probe, but it is slower than the existing four-kernel reference block. Do not
promote this atomic/reset fusion design into the production
`COLLECT_K_BOUNDED` path.

## Scope

- Pod: `root@157.157.221.29 -p 22942`
- Commit base while testing: `017313df` plus local Goal 1565 diagnostic edits
- GPU: RTX 4000 Ada pod environment already used for Goal 1555-1564 work
- Library: `build/librtdl_optix.so`
- Probe: `scripts/goal1565_v1_5_4_optix_collect_k_fused_materialize_mark_probe.py`
- Reference block: materialize, mark, prefix, compact
- Candidate block: memset marks/block_counts, fused materialize+mark with

## Interpretation

The fused kernel preserves parity in this diagnostic, including cross-segment
duplicates generated by identical sorted input segments. However, the cost of
stream resets plus atomic block-count accumulation outweighs the saved mark
kernel launch. This validates Goal 1564's correctness plan but rejects it as a
performance path.

## Next Direction

The next launch-reduction candidate should not use input-row atomics for
`block_counts`. Two options remain:

- output-indexed merge-path materialize+mark, which keeps block-local
  reductions but requires a more complex merge-position-to-input-row mapping;
- stop compact-level fusion and move to a different bottleneck if profiling
  shows merge launch overhead is no longer the dominant recoverable cost.

This result is diagnostic only and does not authorize public speedup wording.
