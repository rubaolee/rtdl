### Q2 — Correctness and ABI risks

**Overall: sound, with two notes.**

**Device-side semantics are correct.** The two kernels are generated by text-substitution specialisation from the base hit-count kernel (`kRayHitCount3DKernelSrc`):

- **Weighted-sum path** starts from `ray_anyhit_kernel_source_3d()`, which rewrites `__anyhit__` to `optixSetPayload_1(1u); optixTerminateRay()`. Payload `p1 ∈ {0,1}`. Final write `if (p1 != 0u) atomicAdd(weighted_hit_sum, ray_weights[idx])` correctly accumulates weights for rays that hit. ✓
- **Hit-count-sum path** starts from the raw `kRayHitCount3DKernelSrc`, which accumulates `p1++` on every hit via `optixIgnoreIntersection()`. Final write `if (p1 != 0u) atomicAdd(hit_count_sum, p1)` correctly folds per-ray counts into a scalar. ✓

**Host/device struct layout alignment is correct.** The text substitutions replace *only* the output field in the device struct, leaving `traversable`, `rays`, `triangles`, and `ray_count` intact. The host-side `RayAnyHitWeightedSum3DLaunchParams` and `RayHitCount3DSumLaunchParams` mirror these layouts exactly (field-for-field). ✓

**ctypes argtypes are bound correctly.** `_find_optional_backend_symbol` calls `getattr(lib, name)`, and ctypes' `CDLL.__getattr__` caches the result on the lib object via `setattr`. So the registration in `_load_and_bind_library()` and the call-time lookup in the method both return the same `_FuncPtr` instance with registered argtypes. ✓

**`RtdlRay3D` / `RtdlTriangle3D` under `#pragma pack(push, 1)` is pre-existing** — these structs and their Python ctypes mirrors are unchanged. The new paths inherit their correctness from the grouped-segment path that already exercises them. ✓

**`atomicAdd` on `unsigned long long`** is architecturally sound on all SM ≥ 3.5 targets. Sum zero-initialisation before launch is present in both paths. ✓

**Note 1 — kernel text-substitution fragility.** If the base kernel source is ever refactored, the substitution guards (`if (pos == std::string::npos) throw ...`) will catch the mismatch at runtime. There is no compile-time protection. This is a pre-existing pattern risk, not introduced here, but it is the one structural fragility in these new paths. No fix required for this PR; document the invariant in a comment if the base kernel is ever touched.

**Note 2 — uint32 payload overflow at extreme degree.** `p1` is a 32-bit OptiX payload. For the hit-count-sum path, a single ray hitting > 2³² triangles would silently wrap. At K4 × 10,000 (max degree 3 → max hits per ray = 3), this is not reachable. Pre-existing limitation, not new.
