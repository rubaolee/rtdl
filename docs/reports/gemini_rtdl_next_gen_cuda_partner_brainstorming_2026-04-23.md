# RTDL Next-Generation Brainstorming: The CUDA Core Partner

**Date:** 2026-04-23
**Topic:** Separating RT Core execution from CUDA Core execution, and finding a Python-native "partner" library for complex mathematical reductions.

## The Core Problem: Hardware Specialization
Modern GPUs contain two vastly different types of execution units:
1.  **RT Cores (Ray Tracing):** Highly specialized, fixed-function hardware designed specifically to accelerate Bounding Volume Hierarchy (BVH) traversal and spatial intersections. This is where RTDL excels.
2.  **CUDA Cores (Streaming Multiprocessors):** General-purpose units designed for dense math, tensor operations, array formatting, and algorithmic reductions.

Real-world applications (like physics simulations, clustering, or graph analytics) require *both*. Currently, RTDL's OptiX backend utilizes CUDA cores internally for the `Refine` and `Emit` phases (OptiX any-hit and closest-hit programs), but this is locked away in pre-compiled C++ binaries. Python users cannot easily inject custom, arbitrary math or logic into the GPU pipeline without dropping down to write raw CUDA/C++.

## The Current Workaround: Hardcoded "Stand-In" Partners
Because RTDL currently lacks a generic Python-native partner (like Triton or Numba) to handle the post-intersection logic, the developers manually wrote bespoke, highly-optimized C++ functions directly into the engine's boundary.

These application-specific "engine boosting" functions (such as `rt.directed_hausdorff_2d_embree` or `rt.prepare_embree_db_dataset`) are essentially doing the exact job a Triton or Numba kernel would do:
1. They take the raw spatial hits generated by the RT Core/BVH.
2. They instantly apply the "boosting" logic (e.g., the `max()` reduction for Hausdorff, or `group_by` aggregations for Database analytics).
3. They return the final, compact payload back to Python.

These "leaky" engine paths are not design flaws; they are manually crafted, highly-optimized stand-ins. In the vNext vision, by introducing a true partner like Triton or Numba, RTDL can delete these hardcoded paths from the core engine, restoring abstraction purity while maintaining massive performance.

## The "Partner" Vision
For the next generation of RTDL, the ideal architecture would enforce a strict separation of concerns:
*   **RTDL acts exclusively as the "RT Core Engine":** The user defines the geometry, and RTDL builds the BVH and runs the spatial queries. Instead of returning millions of rows to the CPU, RTDL outputs the raw collision candidate pointers directly into a GPU device tensor.
*   **A Partner Library acts as the "CUDA Core Engine":** The user writes a custom Python function using a partner library. This library reads the device tensor of collision candidates, applies the business logic, performs the heavy math, and executes the final reductions using standard CUDA cores.

This allows Python developers to achieve 100% maximum hardware utilization (maxing out RT Cores for search, and CUDA Cores for math) without ever leaving Python.

## Evaluating Potential Partners

When selecting a partner library for the CUDA Core execution, there are three main contenders in the Python ecosystem, each suited to different algorithmic paradigms:

### 1. OpenAI Triton
**Best for:** Dense math, block-level aggregations, and data formatting.
*   **The Paradigm:** Triton forces developers to write code from the perspective of a *block of data* rather than a single thread. It automatically manages shared memory and thread assignments for the block.
*   **Pros:** Incredible performance for algorithms that involve heavy memory loading followed by dense math (e.g., summing up massive arrays of intersection distances for a physics simulation).
*   **Cons:** Because it abstracts away individual threads, Triton struggles with highly irregular, branching algorithms where every piece of data takes a wildly different path.

### 2. CuPy
**Best for:** Rapid array operations and users willing to write C++ strings for custom kernels.
*   **The Paradigm:** CuPy is a near-exact drop-in replacement for NumPy (`import cupy as cp`).
*   **Pros:** It has massive mindshare. If the post-intersection logic can be expressed as standard array math, CuPy will execute it flawlessly on the GPU. It also integrates perfectly with PyTorch.
*   **Cons:** If true custom thread-level logic is needed, CuPy requires the user to write `cp.RawKernel` functions, which means writing actual C++ string code inside the Python file.

### 3. Numba
**Best for:** Complex, highly irregular thread-level logic where the user refuses to write C++.
*   **The Paradigm:** Numba uses a Just-In-Time (JIT) compiler (`@cuda.jit`) that translates pure Python syntax directly into PTX (GPU assembly code).
*   **Pros:** Numba exposes the raw CUDA SIMT (Single Instruction, Multiple Thread) model directly to Python. It allows users to manage individual thread IDs, write complex branching logic, and use thread-barriers, all natively in Python.
*   **Cons:** Steeper learning curve for users who don't understand the CUDA execution model, and lacks the automatic block-optimization magic of Triton.

## Conclusion for vNext
If RTDL vNext aims to support highly irregular post-intersection logic (like walking down a complex graph structure based on ray-hit results), **Numba** provides the most flexible thread-level control natively in Python. However, if the goal is to perform massive mathematical reductions (like aggregating point densities), **Triton** or **CuPy** would be the ideal integration target.
