# Goal 130 Report: v0.2 Test Plan

Date: 2026-04-06
Status: accepted

## Summary

Goal 130 starts the v0.2 test-and-execution closure line.

The plan is being built around the current real v0.2 surface:

- `segment_polygon_hitcount`
- `segment_polygon_anyhit_rows`
- the narrow generate-only workflow

The intent is not to create a vague QA wish list. The plan must be:

- repo-accurate
- command-accurate
- tied to the actual v0.2 goals already closed

The accepted plan is based on a Claude-authored draft, then narrowed and
corrected by repo review before execution.

## Accepted test layers

### 1. DSL and schema surface

- language and lowering tests for the two v0.2 workload families
- baseline-contract coverage
- generate-only request validation and payload-shape coverage

Representative modules:

- `tests.goal10_workloads_test`
- `tests.baseline_contracts_test`
- `tests.rtdsl_language_test`
- `tests.test_core_quality`

### 2. Workload-family closure tests

For `segment_polygon_hitcount`:

- semantics tests
- backend runner tests
- closure tests
- backend-audit coverage

Representative modules:

- `tests.goal110_segment_polygon_hitcount_semantics_test`
- `tests.goal110_baseline_runner_backend_test`
- `tests.goal110_segment_polygon_hitcount_closure_test`
- `tests.goal116_segment_polygon_backend_audit_test`

For `segment_polygon_anyhit_rows`:

- PostGIS/parity harness tests
- large-row harness tests

Representative modules:

- `tests.goal128_segment_polygon_anyhit_postgis_test`

### 3. Generate-only workflow tests

The accepted plan keeps this narrow and explicit:

- generate-only is required for the v0.2 workflow surface
- but generate-only backend support is not identical to the full v0.2 backend
  surface

Representative modules:

- `tests.goal111_generate_only_mvp_test`
- `tests.goal113_generate_only_maturation_test`
- `tests.goal129_generate_only_second_workload_test`

### 4. Local execution matrix

The accepted repo runner now exposes a clean local v0.2 group:

- `python3 scripts/run_test_matrix.py --group v0_2_local`

That group is designed to stay green on a developer machine even when Linux-only
native rows are not available.

### 5. Linux/PostGIS execution matrix

The accepted Linux rows remain environment-gated and should not be overclaimed
as universal local requirements.

Required Linux/PostGIS-backed checks:

- `segment_polygon_hitcount` large deterministic rows
- `segment_polygon_anyhit_rows` large deterministic rows
- correctness against PostGIS
- backend timing comparison for:
  - `cpu`
  - `embree`
  - `optix`
  - `vulkan`

## Known corrections applied before acceptance

The initial draft required corrections:

- remove nonexistent `tests.plan_schema_test`
- explicitly include Goal 116 backend-audit coverage
- distinguish generate-only backend support from the broader v0.2 backend
  surface
- treat Vulkan as environment-gated rather than universal local execution
- repair the stale canonical runner surface by adding:
  - `v0_2_local`
  - `v0_2_linux`
  - `v0_2_full`

## Acceptance rule

The plan is accepted only if:

- the named test files and commands are real
- the Linux-only rows are called out honestly
- unsupported prepared modes are not rendered as fake zero timings
- the package can produce one practical local v0.2 test result and one
  Linux/PostGIS-backed large-scale result
