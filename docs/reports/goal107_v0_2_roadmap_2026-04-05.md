# Goal 107 RTDL v0.2 Roadmap Proposal

Date: 2026-04-05
Author: Codex
Status: complete

## Purpose

This report proposes the next major RTDL phase after the v0.1 release.

It is intentionally strategic rather than experimental. The point is to decide
where the next real engineering effort should go.

## Starting point

RTDL v0.1 already established:

- a bounded reviewed research release
- one real language/runtime surface
- Embree and OptiX as serious performance backends
- Vulkan as a correct portable backend
- a release-facing documentation, validation, and audit trail

That means the next phase should not be another v0.1-style cleanup loop.

## Proposed v0.2 direction

The first draft proposed three co-equal pillars. The adversarial reviews
correctly rejected that as too diffuse.

The revised v0.2 direction is:

1. one release-defining bet:
   - broaden RTDL with one additional workload family beyond the v0.1
     RayJoin-centered slice
2. one constrained secondary bet:
   - pure code-generation mode for one accepted kernel family only
3. one supporting track:
   - further performance maturation on real NVIDIA RT cores, with Embree kept
     strong and Vulkan kept honest

## Primary user

The primary v0.2 user is:

- an advanced technical user, research engineer, or systems programmer who
  wants to express one non-graphical ray-tracing workload without first
  hand-writing separate backend-specific implementations

## Why this is the right next step

### Broader workloads

If RTDL remains only a RayJoin-centered artifact, it will remain intellectually
interesting but strategically narrow.

v0.2 should prove that RTDL can support a broader class of non-graphical
ray-tracing workloads, but still within a disciplined scope.

### Code-generation mode

If RTDL can only run inside its own current environment, its product value is
limited.

A generate-only mode would let RTDL act as:

- a runtime system
- a code authoring system

That makes the project more useful to users who want code output rather than
runtime execution.

### Performance maturation

The strongest present performance path is still real RT hardware, especially
OptiX on NVIDIA GPUs.

So v0.2 should keep improving the path that already has evidence instead of
pretending there is equal maturity on hardware we do not have.

## Recommended v0.2 scope discipline

### In core scope

- additional RTDL workload families that still fit candidate-generation plus
  refine structure
- one constrained generate-only mode with real runnable output
- OptiX/Embree performance tuning on accepted hardware and workload surfaces

### Experimental but not core

- more ambitious ray-query kernels that may or may not generalize
- broader non-join demo workloads
- Vulkan improvements that strengthen coverage or portability

### Out of scope for now

- native AMD backend promises without AMD hardware
- native Intel backend promises without Intel GPU hardware
- distributed execution
- full exact polygon overlay materialization

## Hard gates

The adversarial reviews made one thing clear: v0.2 needs explicit kill
criteria, not only ambitions.

Required gates:

- Goal 108 must classify workloads before expansion starts
- Goal 109 must be cut or paused if it does not prove non-toy value
- Goal 112 must not dominate the release unless it directly supports the
  chosen new workload family or removes a release blocker

## Recommended first-wave goals

1. Goal 108: v0.2 workload scope charter
2. Goal 110: one additional workload-family closure
3. Goal 109: generate-only mode MVP
4. Goal 111: generated-code verification harness
5. Goal 112: OptiX/Embree performance maturation

## Final recommendation

RTDL v0.2 is promising **if and only if** it stays disciplined:

- broader than RayJoin
- stricter than “support everything”
- honest about hardware constraints
- serious about generated code quality

The strongest correction from the aggressive reviews is this:

- **v0.2 must be workload-first**
- code generation is allowed only as a constrained second bet

That is the roadmap this goal recommends.
